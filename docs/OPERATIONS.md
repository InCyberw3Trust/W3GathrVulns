## Requirements

- Docker Engine >= 24
- Docker Compose v2 (`docker compose`, not `docker-compose`)
- `openssl` (for certificate generation, available on most systems)
- Ports 80 and 443 available on the host

## First Installation

```bash
git clone <repo-url>
cd w3gathrvulns/tool

# Run the interactive setup — generates .env with secure secrets
bash setup.sh
```

`setup.sh` will ask for:
- **SERVER_IP**: the IP or hostname your team will use to access the instance (used in the TLS certificate and CORS config)
- **HTTP_PORT** (optional, default 80)
- **HTTPS_PORT** (optional, default 443)
- **Admin password**: the password for the web UI (`admin` username)

It generates a `.env` file with a random `POSTGRES_PASSWORD` (32 chars), `SECRET_KEY` (64 chars), `API_TOKEN_READ`, and `API_TOKEN_WRITE`, all with permissions `600`. The file is git-ignored.

Then start the stack:

```bash
docker compose up -d --build
```

First startup takes ~1–2 minutes: certificate generation, database initialization, and application boot all happen sequentially.

The UI is available at `https://<SERVER_IP>`. Your browser will warn about the self-signed certificate — this is expected. Accept the exception or replace the certificate with one from a trusted CA.

The API documentation is available at `https://<SERVER_IP>/api/docs`.

## Environment Variables

All variables are set in `tool/.env`. Never commit this file.

| Variable | Default | Required | Description |
|---|---|---|---|
| `SERVER_IP` | — | Yes | Hostname or IP for the TLS certificate CN and CORS |
| `HTTP_PORT` | `80` | No | Host port for HTTP (redirects to HTTPS) |
| `HTTPS_PORT` | `443` | No | Host port for HTTPS |
| `POSTGRES_USER` | `w3gathrvulns` | No | PostgreSQL user |
| `POSTGRES_PASSWORD` | — | Yes | PostgreSQL password (generated by setup.sh) |
| `POSTGRES_DB` | `w3gathrvulns` | No | PostgreSQL database name |
| `SECRET_KEY` | — | Yes | Cryptographic key for JWT signing, minimum 32 characters |
| `CORS_ORIGINS` | `["https://SERVER_IP"]` | Yes | JSON array of allowed origins |
| `DEBUG` | `false` | No | Enable verbose logging (do not use in production) |
| `UI_USERNAME` | `admin` | No | Username for the web UI login |
| `UI_PASSWORD` | — | Yes | Password for the web UI login (set during setup.sh) |
| `API_TOKEN_READ` | — | Yes | Bearer token for read-only API access (GET requests) |
| `API_TOKEN_WRITE` | — | Yes | Bearer token for full API access (GET + POST + PATCH + DELETE) |
| `JWT_EXPIRE_HOURS` | `24` | No | Duration of web UI sessions in hours |
| `DEMO_MODE` | `false` | No | When `true`: disables token/password changes and resets all data hourly |

The `DATABASE_URL` is automatically derived from the Postgres variables — do not set it manually.

## Authentication

### Web UI

Open `https://<SERVER_IP>` — you are redirected to the login page. Enter the credentials configured as `UI_USERNAME` / `UI_PASSWORD` in `.env`. The session is a JWT valid for `JWT_EXPIRE_HOURS` hours (default 24). Clicking **Sign out** in the sidebar invalidates the session immediately.

### API Tokens (CI/CD integrations)

All API endpoints require a `Bearer` token in the `Authorization` header. Two tokens are available:

| Token | Variable | Allowed methods |
|---|---|---|
| Read token | `API_TOKEN_READ` | `GET` only — dashboards, exports, queries |
| Write token | `API_TOKEN_WRITE` | All methods — ingest, findings updates, rules |

**Usage:**
```bash
# Ingest (requires write token)
curl -k -X POST "https://<SERVER_IP>/api/v1/ingest/trivy?project=myapp" \
  -H "Authorization: Bearer $API_TOKEN_WRITE" \
  -H "Content-Type: application/json" -d @trivy.json

# Read dashboard stats (works with read or write token)
curl -k "https://<SERVER_IP>/api/v1/stats/dashboard" \
  -H "Authorization: Bearer $API_TOKEN_READ"
```

The `API_TOKEN_READ` and `API_TOKEN_WRITE` values are printed by `setup.sh` and stored in `.env` as the initial seed. After first startup they are imported into the database and the DB is authoritative.

**To rotate tokens**: go to **Settings** → API Tokens → click **Regenerate** on the token you want to rotate. The new token is shown once — copy it immediately and update any CI/CD pipelines using it. No restart required.

**To change the admin password**: go to **Settings** → Admin Password → enter current and new password.

## Integrating Scanners

All ingest endpoints accept a `POST` request with the scanner's JSON output as the body. All ingest requests require the **write token** (`Authorization: Bearer $API_TOKEN_WRITE`).

**Base URL**: `https://<SERVER_IP>/api/v1/ingest`

**Common query parameters**:

| Parameter | Required | Supported endpoints | Description |
|---|---|---|---|
| `project` | Yes | All | Project name — created automatically if it does not exist |
| `branch` | No | All | Git branch name |
| `commit` | No | All | Git commit SHA |
| `pipeline` | No | All | CI/CD pipeline ID |
| `repo_url` | No | `/trivy`, `/gitlab-sast` | Repository URL (enables clickable file links in the UI) |
| `scan_date` | No | All | ISO 8601 datetime to override `first_seen` / `last_seen` (useful for backfilling historical data) |

### Trivy

```bash
trivy image --format json --output trivy.json myimage:latest
curl -k -X POST "https://<SERVER_IP>/api/v1/ingest/trivy?project=my-project&branch=main&commit=$GIT_SHA" \
  -H "Authorization: Bearer $API_TOKEN_WRITE" \
  -H "Content-Type: application/json" \
  -d @trivy.json
```

GitLab CI example:
```yaml
trivy-scan:
  image: aquasec/trivy:latest
  script:
    - trivy image --format json --output trivy.json $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA
    - |
      curl -k -X POST "https://<SERVER_IP>/api/v1/ingest/trivy?project=$CI_PROJECT_NAME&branch=$CI_COMMIT_REF_NAME&commit=$CI_COMMIT_SHA&pipeline=$CI_PIPELINE_ID" \
        -H "Authorization: Bearer $API_TOKEN_WRITE" \
        -H "Content-Type: application/json" -d @trivy.json
```

### OWASP ZAP

```bash
docker run --rm ghcr.io/zaproxy/zaproxy:stable \
  zap-baseline.py -t https://target.example.com -J /tmp/zap.json || true

curl -k -X POST "https://<SERVER_IP>/api/v1/ingest/owasp-zap?project=my-app&branch=main" \
  -H "Authorization: Bearer $API_TOKEN_WRITE" \
```

### Nuclei

Nuclei outputs JSONL (one JSON object per line). Wrap it into a JSON array before sending:

```bash
nuclei -u https://target.com -jsonl -o results.jsonl
jq -s '.' results.jsonl > results.json

curl -k -X POST "https://<SERVER_IP>/api/v1/ingest/nuclei?project=my-app" \
  -H "Authorization: Bearer $API_TOKEN_WRITE" \
  -H "Content-Type: application/json" \
  -d @results.json
```

## Auto-Triage Rules

Rules automatically change the `status` or `severity` of findings on ingest. They are managed from the **Rules** page in the UI.

### Rule Structure

A rule has:
- **Conditions**: one or more field/operator/value checks
- **Conditions mode**: `all` (AND) or `any` (OR)
- **Actions**: `set_status` or `set_severity`
- **Priority**: lower number = higher priority (rules applied ascending)

**Available fields**: `title`, `source`, `severity`, `status`, `vuln_id`, `cve`, `file_path`, `component`, `tags`, `description`

**Operators**: `equals`, `not_equals`, `contains`, `not_contains`, `starts_with`, `ends_with`, `in`, `regex`

### Examples

Mark all Nuclei INFO findings as closed:
```json
{
  "conditions": [
    {"field": "source", "operator": "equals", "value": "nuclei"},
    {"field": "severity", "operator": "equals", "value": "INFO"}
  ],
  "conditions_mode": "all",
  "actions": [{"type": "set_status", "value": "CLOSED"}]
}
```

Auto-accept known false positive CVE:
```json
{
  "conditions": [
    {"field": "cve", "operator": "equals", "value": "CVE-2023-12345"}
  ],
  "conditions_mode": "all",
  "actions": [{"type": "set_status", "value": "FALSE_POSITIVE"}]
}
```

### Cron Scheduling

Each rule has an optional `cron_schedule` field (standard 5-field cron, e.g. `0 * * * *` for every hour). When set, the built-in APScheduler executes the rule automatically on schedule — no external cron setup required.

The schedule is registered instantly when you create or update a rule. You can also trigger execution manually at any time:

```bash
# Trigger all scheduled rules now
curl -k -X POST "https://<SERVER_IP>/api/v1/rules/run-scheduled"
```

### Applying Rules in Bulk

Rules are applied automatically on every new ingest. To re-apply all rules to existing findings:

```bash
curl -k -X POST "https://<SERVER_IP>/api/v1/rules/apply-all"
```

Or use the **Apply all rules** button in the Rules page.

### Import / Export

Rules can be exported to JSON and imported on another instance (Rules page → Export / Import). Useful for sharing rule sets between environments or teams.

## Export and Reporting

From the Findings page, use the **Export CSV** or **Export PDF** buttons. Exports respect the active filters (project, severity, status, source, search query).

Via API:

```bash
# CSV export for all CRITICAL and HIGH open findings in project "my-app"
curl -k "https://<SERVER_IP>/api/v1/export/csv?project_id=<uuid>&severity=CRITICAL&severity=HIGH&status=OPEN" \
  -H "Authorization: Bearer $API_TOKEN_READ" \
  -o findings-export.csv

# PDF report
curl -k "https://<SERVER_IP>/api/v1/export/pdf?project_id=<uuid>" \
  -H "Authorization: Bearer $API_TOKEN_READ" \
  -o findings-report.pdf
```

PDF export is capped at 500 findings per report.

## Backup and Restore

### Backup

```bash
cd tool/
docker compose exec db pg_dump -U w3gathrvulns w3gathrvulns | gzip > backup-$(date +%Y%m%d-%H%M%S).sql.gz
```

Store the backup file outside the Docker volume. Also back up your `tool/.env` file (keep it secure — it contains secrets).

### Restore

```bash
cd tool/
# Stop the backend to avoid writes during restore
docker compose stop backend

# Restore
gunzip -c backup-20240114-120000.sql.gz | docker compose exec -T db psql -U w3gathrvulns w3gathrvulns

# Restart
docker compose start backend
```

## Maintenance

### Updating

```bash
cd w3gathrvulns/
git pull
cd tool/
docker compose up -d --build
```

Database migrations run automatically on startup — no manual steps required.

### Rotating Secrets

Different credentials are managed in different places:

**API tokens and admin password** are stored in the database and managed exclusively from the **Settings page** in the UI. Changes take effect immediately without any restart. Do not edit `.env` for these — it is not read after first startup.

**`SECRET_KEY`** is read from `.env` at container startup. To rotate it:

```bash
cd tool/
bash setup.sh --rotate-secrets   # rewrites SECRET_KEY in .env
docker compose up -d --build backend
```

This invalidates all active web UI sessions (users will need to log in again).

**`POSTGRES_PASSWORD`** requires updating the running PostgreSQL instance separately. To rotate without data loss:

```bash
# 1. Update the password in the running database
docker compose exec db psql -U w3gathrvulns -c "ALTER USER w3gathrvulns WITH PASSWORD 'newpassword'"

# 2. Update .env with the same value
#    Edit POSTGRES_PASSWORD= in tool/.env

# 3. Restart
docker compose up -d --build
```

### Resetting All Data

```bash
curl -k -X DELETE "https://<SERVER_IP>/api/v1/ingest/reset?confirm=YES_DELETE_EVERYTHING"
```

This permanently deletes all findings, scans, and projects. It cannot be undone.

### Disk Usage

The PostgreSQL volume (`postgres_data`) grows with the number of findings and raw scan payloads stored. Check volume size:

```bash
docker system df -v | grep postgres_data
```

Raw scan payloads (`raw_payload` column in `scans`) are the largest contributor. There is currently no automatic retention policy — clean up via the UI (delete projects) or direct SQL if needed.

## Demo Mode

Demo mode is designed for public demonstration instances. It is activated by `DEMO_MODE=true` in `tool/.env`.

### Enabling demo mode

The easiest way is to answer **y** to the "Enable demo mode?" prompt during `bash setup.sh`. This sets `DEMO_MODE=true` and `UI_PASSWORD=W3Gathr!Demo` in the generated `.env`.

To enable it manually on an existing install, set `DEMO_MODE=true` in `tool/.env` and restart the backend:

```bash
docker compose up -d --build backend
```

### What demo mode does

| Behaviour | Detail |
|---|---|
| **Hourly data reset** | On startup and every hour at `:00 UTC`, all findings, scans, and projects are wiped and replaced with sample data across 3 projects (`web-frontend`, `backend-api`, `infra`) using all 6 supported scanners, spread over 30 days |
| **Mutations blocked** | `POST /settings/regenerate-token` and `POST /settings/change-password` return `HTTP 403 Not available in demo mode` — enforced at the API level |
| **UI locked** | The Settings page shows a warning banner and replaces the token/password forms with a locked notice |
| **Read/write access** | All other features (dashboard, findings, rules, export) remain fully functional |

### Demo credentials

| Field | Value |
|---|---|
| Username | `admin` |
| Password | `W3Gathr!Demo` |

### Disabling demo mode

Set `DEMO_MODE=false` in `tool/.env` and restart: `docker compose up -d --build backend`.

## Running Tests

Tests use an **in-memory SQLite database** — no PostgreSQL or Docker required.

```bash
cd tool/backend
python -m venv .venv && source .venv/bin/activate
# requirements-test.txt excludes psycopg2/pandas — works on Python 3.12 and 3.13
pip install -r requirements-test.txt
pytest
```

Verbose output:

```bash
pytest -v
```

Run a specific file:

```bash
pytest tests/test_parsers.py -v
pytest tests/test_auth.py::TestLogin::test_valid_login_returns_token -v
```

| File | What it covers |
|---|---|
| `tests/conftest.py` | SQLite DB setup, authenticated test clients |
| `tests/test_parsers.py` | All scanner parsers (Trivy, GitLab SAST, ZAP, Nuclei) |

## Troubleshooting

### Containers not starting

```bash
cd tool/
docker compose logs
docker compose ps
```

Common causes:
- **Missing `.env`**: run `bash setup.sh`
- **Port already in use**: change `HTTP_PORT` or `HTTPS_PORT` in `.env`
- **Database not ready**: the backend waits for PostgreSQL health check; if the DB never becomes healthy, check `docker compose logs db`

### Backend crashes on startup

```bash
docker compose logs backend
```

Common causes:
- `SECRET_KEY` is missing, too short (< 32 chars), or set to a known insecure value — the app will refuse to start with a clear error message
- `DATABASE_URL` is malformed

### Ingest returns 422

The payload format is invalid for the selected scanner. Check:
- You are sending to the correct endpoint for your scanner
- The payload is valid JSON
- For Nuclei: the output must be a JSON array (run `jq -s '.' nuclei.jsonl > nuclei.json`)

Internal parse errors are logged server-side:
```bash
docker compose logs backend | grep "ingest error"
```

### Swagger UI shows "Unable to render this definition"

This happens when the OpenAPI schema URL is not proxied through Nginx. Verify that `openapi_url="/api/openapi.json"` is set in `tool/backend/app/main.py`. It should be present in this version.

### Frontend shows blank page or routing errors

```bash
docker compose logs frontend
```

The Nginx `try_files` fallback handles client-side routing — all routes should return `index.html`. If you see 404s, check the nginx.conf is correctly mounted in the frontend container.

### Rules are not being applied

- Confirm the rule is **enabled** (toggle in the Rules page)
- Use the **Test** button to preview which findings would match
- Check rule **priority** — lower number runs first
- Re-apply manually: Rules page → **Apply all rules**

## Known Limitations

| Area | Limitation |
|---|---|
| **Authentication** | Single admin account (username/password). No multi-user or RBAC support. |
| **Rate limiting** | No rate limiting on API endpoints. |
| **TLS** | Self-signed certificate by default — browsers will warn. Replace with a CA-signed cert for team use. |
| **Ingest payload size** | Maximum 50 MB per request (Nginx limit). Very large scan reports should be split. |
| **Ingest timeout** | 120 seconds proxy timeout. Parsers for very large reports may time out. |
| **PDF export** | Capped at 500 findings per PDF. |
| **Horizontal scaling** | Single PostgreSQL instance, no sharding. Suitable for small to medium teams. |
| **Data retention** | No automatic purging. Delete old projects manually to reclaim space. |
| **Cron scheduling** | Built-in APScheduler runs rules on their cron schedule. For high-availability deployments with multiple replicas, use an external trigger (`POST /api/v1/rules/run-scheduled`) instead to avoid duplicate execution. |
| **Deduplication** | Race condition possible under very high concurrent ingest volume (multiple pipelines sending simultaneously). |
